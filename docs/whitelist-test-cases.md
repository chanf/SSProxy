# 豆豉代理 (luci-app-mihomo) 分流与服务控制测试用例

本文档设计了针对**豆豉代理**（原 `luci-app-mihomo`）的完整网络功能与状态测试用例。测试逻辑将系统状态严格分为**「服务关闭状态」**与**「服务运行状态」**两大核心版块，以避免任何用例冲突。

---

## 1. 测试环境基准设置

### 1.1 拓扑基础信息
* **软路由网关 IP**：`192.168.66.1`
* **测试客户端 A (白名单用户)**：IP 地址为 `192.168.66.158`，主机名为 `ThinkFeng`。
* **测试客户端 B (直连直通用户)**：IP 地址为 `192.168.66.100`，主机名为 `StandardDevice`。
* **代理服务器 (节点)**：已导入有效订阅，确保代理节点联通正常。
* **被拦截/墙外域名**：`google.com`（在代理生效下应被代理转发，在无代理/直通下应被墙）。
* **常规国内域名**：`baidu.com`（在任何情况下都应直连放行）。

## 3. 详细测试用例步骤与预期结果

### 📌 第一部分：服务关闭状态测试（物理网络状态验证）

#### 🧪 用例 TC-05：服务关闭状态（停用开关）测试
* **测试目的**：验证当主服务关闭时，所有网络劫持规则均已被物理清理，所有设备回归原始网络，不走任何代理。

##### 1. 前置条件与路由器配置
* 在 Web 界面 Dashboard 点击 **「停止服务」**，或者在路由器 SSH 中运行 `/etc/init.d/mihomo stop`。

##### 2. 测试步骤与判定指标
1. **检查路由器配置与规则**（SSH 内执行）：
   * 运行 `nft list table inet mihomo` 检查防火墙。
     * **预期结果**：提示 `Error: No such file or directory` (即 `inet mihomo` 表已被完全销毁，不占用任何系统资源)。
   * 运行 `uci show dhcp.@dnsmasq[0]` 检查 DNS。
     * **预期结果**：`server` 项没有 `127.0.0.1#1053`，且 `noresolv` 为 `0` 或空（DNS 劫持配置被干净清除）。
2. **验证客户端 A (`.158`) 与客户端 B (`.100`) 的上网状态**：
   * 两台设备上分别查询 DNS：`nslookup google.com`。
     * **预期结果**：均返回普通的公网真实 IP（绝对不能是 `198.18.x.x`）。
   * 两台设备上访问国内网站：`baidu.com`。
     * **预期结果**：均能正常且快速地直接打开（证明本地网络物理联通）。
   * 两台设备上访问被墙网站：`google.com`。
     * **预期结果**：**均无法访问（连接超时或被阻断）**。证明此时所有代理功能已完全关闭，所有设备全部直通 WAN，不经过任何 Mihomo 流程。

---

### 📌 第二部分：服务开启状态测试（分流与接管逻辑验证）

#### 🧪 用例 TC-01：标准白名单直通与代理分流测试（重点用例）
* **测试目的**：验证在不使用 TUN 且不劫持全局 DNS 的前提下，只有白名单用户走代理，非白名单设备完全旁路直通。

##### 1. 前置条件与路由器配置
* 确保服务已开启（`/etc/init.d/mihomo start`）。
* **配置参数**：`tun_enabled=0`，`dns_hijack=0`，`acl_mode='whitelist'`，且 `acl_ips='192.168.66.158'`。

##### 2. 测试步骤与判定指标
1. **检查路由器规则**：
   * 运行 `nft list table inet mihomo`，应当有且仅有一条包含白名单的 return 规则：
     `ip saddr != { 192.168.66.158 } return`。
2. **验证白名单设备 A (`192.168.66.158`)**：
   * 解析 `google.com` 得到真实公网 IP。
   * 访问 `google.com` 正常打开。
   * 查看 Web 界面「访问日志」：发现 `ThinkFeng (192.168.66.158)` 发往 `google.com` 的连接，且该设备在 IP 列表中被**从右侧移至左侧「红杏 (走代理)」**区域。
   * 访问 `baidu.com` 正常直连（Mihomo connections 中显示策略为 `DIRECT`）。
3. **验证非白名单设备 B (`192.168.66.100`)**：
   * 解析 `google.com` 得到真实公网 IP。
   * 访问 `google.com` **连接超时被墙（完全绕过 Mihomo）**。
   * 查看 Web 界面「访问日志」：Mihomo connections 中**没有任何** `.100` 发起的外网请求记录。设备在 IP 列表中**仅留在右侧「设备 (直通/直连)」**区域，不发生任何移动。
   * 访问 `baidu.com` 正常流畅打开（走系统底层默认 WAN 网关直连）。

#### 🧪 用例 TC-02：开启全局 DNS 劫持时的逻辑覆盖测试
* **测试目的**：验证当用户开启全局 DNS 劫持时，白名单自动退化失效，所有设备强行重定向至代理。

##### 1. 前置条件与路由器配置
* **配置参数**：`dns_hijack=1`，前端白名单设置项自动变灰（禁用并锁定为 `all`）。

##### 2. 测试步骤与判定指标
1. 检查 Dnsmasq 配置：`uci show dhcp.@dnsmasq[0]`。
   * **预期结果**：包含 `server='127.0.0.1#1053'` 和 `noresolv='1'`（DNS 被强制劫持到 Mihomo）。
2. 在设备 B (`.100`) 上查询 DNS：`nslookup google.com`。
   * **预期结果**：由于劫持生效，返回 **`198.18.x.x`** 的 Fake-IP。
3. 在设备 B 访问 `google.com`。
   * **预期结果**：可以正常实现代理访问。
4. 查看 Web 界面「访问日志」。
   * **预期结果**：设备 B 的 IP 成功被移至左侧「红杏」区域。证明 DNS 劫持开启时，所有设备全局代理接管，白名单已失效。

---

#### 🧪 用例 TC-03：启用 TUN 模式下的流量接管测试
* **测试目的**：验证开启 TUN 虚拟网卡时，白名单自动失效，全局设备被 TUN 路由接管。

##### 1. 前置条件与路由器配置
* **配置参数**：`tun_enabled=1`，前端白名单自动变灰失效。

##### 2. 测试步骤与判定指标
1. 检查防火墙：运行 `nft list table inet mihomo`。
   * **预期结果**：`inet mihomo` 防火墙表不存在。
2. 检查系统路由表：运行 `ip route show` 确认存在由 TUN 引入的系统默认路由。
3. 在设备 B (`.100`) 上访问 `google.com`。
   * **预期结果**：可以正常通过代理上网（流量走系统 TUN 网卡接管，白名单失效）。

---

#### 🧪 用例 TC-04：全员 TProxy 转发测试 (不限 IP)
* **测试目的**：验证关闭白名单时，使用 TProxy 全局转发。

##### 1. 前置条件与路由器配置
* **配置参数**：`tun_enabled=0`，`dns_hijack=0`，`acl_mode='all'`。

##### 2. 测试步骤与判定指标
1. 检查 nftables：prerouting 链中没有任何 `ip saddr` 的过滤判定规则。
2. 在设备 B (`.100`) 上访问 `google.com`。
   * **预期结果**：可以正常进行代理访问。
