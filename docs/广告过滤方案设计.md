# 水杉代理（luci-app-ssproxy）广告过滤功能 · 方案设计文档

> 本文档对比三种路由器广告过滤技术方案（A/B/C），重点详述选定的**方案 A（Mihomo rule-provider）**的原理、设计与实施步骤。B/C 作为对比与备选简述。本文档为已确认选型并据以编码的实现依据。

---

## 一、背景与目标

水杉代理已是 **Mihomo 单核心 + dnsmasq DNS 劫持 + 受控配置生成**的自洽架构（TProxy 透明代理、fake-ip、订阅自动更新、访问规则管理俱全）。本次需在此基础上增加**广告过滤**能力，目标：

- 拦截主流广告/追踪域名，提升浏览体验、减少无效流量；
- 与现有代理分流深度集成，不引入额外常驻进程；
- 提供 LuCI 图形化管理（开关 + 规则源增删），核心重启即生效；
- 复用项目已有的"受控配置注入"管道，保持单源真相架构。

---

## 二、技术原理：路由器广告过滤的三个拦截层

路由器处于终端流量的必经关口，可在三个阶段识别并丢弃广告：

| 拦截层 | 作用阶段 | 机制 | 代表实现 |
|---|---|---|---|
| **① DNS 层** | 域名解析 | 广告域名直接返回空 IP（黑洞 `0.0.0.0` 或 `rcode://success`），终端拿不到真实地址，连接无从发起 | AdGuard Home、dnsmasq hosts、Mihomo nameserver-policy |
| **② 规则/连接层** | 连接建立 | 终端已解析出 IP 并发起连接，代理/防火墙按规则匹配后 REJECT/DROP | Mihomo rules REJECT、nftables |
| **③ 网络/IP 层** | 包转发 | nftables 直接按 IP-CIDR DROP 广告服务器 IP 段 | nftables 黑名单 |

**本项目架构映射**：Mihomo 同时承担 ①（fake-ip DNS）和 ②（规则引擎），dnsmasq 做 DNS 劫持转发给 Mihomo，nftables 做 TProxy。因此方案 A 能在 ①② 两层同时发力，是架构上最自然的选择。

> fake-ip 模式下，终端拿到的本就是"假 IP"，真实解析被延迟到连接阶段。因此 ① 层（nameserver-policy 在解析阶段就返回空）能比普通 DNS 黑洞更早地"连假 IP 都不分配"，对硬编码域名/IP 混合的 App 广告 SDK 更彻底。

## 三、方案对比总览

| 维度 | A. Mihomo rule-provider ⭐ | B. AdGuard Home | C. dnsmasq + hosts |
|---|---|---|---|
| 拦截层 | ① DNS + ② 规则（双重） | ① DNS | ① DNS |
| 额外进程 | 无 | 独立服务 | 无（复用 dnsmasq） |
| 内存占用 | ≈0（核心已运行） | +30~50 MB | dnsmasq 加载大列表后上涨 |
| 与现有架构 | 完美契合 | 与 DNS 劫持抢 53 端口，三方纠缠 | 可挂 dns_hijack 钩子 |
| 规则智能性 | 高（域名/后缀/关键字/IPCIDR/GEOSITE、白名单优先级） | 中（纯域名） | 低（精确/后缀） |
| 代理分流共存 | 原生支持 | 正交（需额外协调） | 无 |
| 规则更新 | 核心自带 interval | 自带 | 需自写 cron |
| 前端集成 | 复用现有 settings/rules 模式 | 需独立管理界面 | 复用 dns_hijack 开关 |
| **结论** | **选定** | 否决（违背单核心设计） | 仅作低配备选 |

## 四、方案 A：Mihomo rule-provider（选定方案，详述）

### 4.1 原理

#### 4.1.1 Mihomo rule-provider 机制

`rule-providers` 是 Mihomo（Clash Meta）的"规则集订阅"能力：把一大段外部规则（如 anti-AD 的十几万条广告域名）打包成一个**具名规则集**，核心按配置自动下载、缓存、周期更新，再在 `rules` 里用一行引用即可批量匹配。关键字段：

```yaml
rule-providers:
  ssproxy-ad-antiad:
    type: http              # http 远程订阅 / file 本地文件
    behavior: domain        # domain | ipcidr | classical（决定能匹配什么、能否用于 DNS 层）
    format: yaml            # yaml | text
    url: https://.../anti-ad-clash.yaml
    path: ./ruleset/ssproxy-ad-antiad.yaml   # 必须在核心 -d 工作目录内
    interval: 86400         # 秒，自动周期更新

rules:
  - RULE-SET,ssproxy-ad-antiad,REJECT        # 一行引用整个规则集
```

**两个硬约束**（实现时必须遵守）：

1. **`path` 必须在核心 `-d` 工作目录内**。本项目核心以 `-d /etc/mihomo` 启动（[build_ipk.py:205](build_ipk.py#L205)），故 path 只能用 `./ruleset/xxx.yaml`（相对 work_dir）。这是 Mihomo 的安全限制。
2. **`RULE-SET` 引用必须先有对应 `rule-providers` 定义**，否则核心启动 fatal-exit。现有代码 [emit_access_rules_yaml](build_ipk.py#L674-L677) 的注释专门警告过此坑。

#### 4.1.2 behavior 与双层拦截的关系

- `behavior: domain` —— 只匹配域名，**可被 DNS 层 `nameserver-policy` 的 `rule-set` 引用**；
- `behavior: classical` —— 支持复杂规则（含 IP/端口等），但**只有 domain 部分能用于 DNS 层**；
- `behavior: ipcidr` —— 只匹配 IP，仅用于规则层。

因此方案 A 要求：**参与 DNS 层拦截的源必须是 domain behavior**；非 domain 源（如某些 classical 综合规则集）只进规则层、不进 nameserver-policy。UI 会据此区分。

#### 4.1.3 双重拦截在 fake-ip 架构下的工作流程

```
终端查询 ads.example.com
        │
        ▼
  dnsmasq (DNS 劫持) ──转发──► Mihomo DNS (fake-ip, :1053)
        │
        ▼
  ┌─ nameserver-policy 命中 "rule-set:ssproxy-ad-*" ?
  │    YES → rcode://success 返回空   【① DNS 层拦截：连假 IP 都不分配】
  │    NO  → 分配 fake-ip，返回终端
  └─ 终端用 fake-ip 发起 TCP/UDP 连接
        │
        ▼
  Mihomo 规则引擎匹配 rules
  ┌─ RULE-SET,ssproxy-ad-*,REJECT 命中 ?
  │    YES → 连接被 REJECT            【② 规则层拦截：兜底】
  │    NO  → 按其他规则分流/直连
```

两层互补：① 层更早、更省资源（解析即拦，不发连接）；② 层兜底（即便 ① 未命中，连接阶段仍可拦）。这正是用户选定的"双重拦截"。

### 4.2 方案设计

#### 4.2.1 UCI 数据模型

在 `config mihomo 'config'` 段（[build_ipk.py:55-78](build_ipk.py#L55-L78)）**仅加一个 option**：

```
option adblock_enabled '0'      # 广告过滤总开关
```

新增 `adblock_source` 段类型（每个规则源一段，运行时 `uci add` 动态产生，仿 [add_access_rule:1576](build_ipk.py#L1576)）：

| 字段 | 含义 | 示例 |
|---|---|---|
| `enabled` | 是否启用 | `1` |
| `name` | provider 名，**强制 `ssproxy-ad-` 前缀**防与订阅重名 | `ssproxy-ad-antiad` |
| `url` | rule-provider 订阅地址 | anti-AD / Loyalsoldier reject 等 |
| `behavior` | `domain` \| `classical`（默认 domain） | `domain` |
| `format` | `yaml` \| `text`（默认 yaml） | `yaml` |

**内置预设源**（anti-AD、Loyalsoldier reject 等两三个，必须 domain behavior）：不写进默认 config，而是在 helper.sh 首次运行时**幂等 `uci add`**（`uci -q show` 判断不存在再 add，与 `mihomo_rule` 范式一致）。`conffiles` 已覆盖 `/etc/config/mihomo`（[build_ipk.py:51](build_ipk.py#L51)），新 option 与新段自动跨升级保留。

#### 4.2.2 helper.sh 后端（[build_ipk.py](build_ipk.py) helper.sh 字符串内）

新增两个"UCI → YAML"产出函数（仿 [emit_access_rules_yaml:673-721](build_ipk.py#L673-L721)）：

- **`emit_adblock_rule_providers_yaml`**：遍历启用的 `adblock_source` 段，产出受控 `rule-providers:` 块，每条含 `type: http` / `behavior` / `format` / `url` / `path: ./ruleset/<name>.yaml` / `interval`。
- **`emit_adblock_rules_yaml`**：产出 `RULE-SET,<name>,REJECT` 行。

新增 CRUD 子命令（挂 case 分发器 [build_ipk.py:1831-1898](build_ipk.py#L1831-L1898)，照抄 [add/del_access_rule:1570-1595](build_ipk.py#L1570-L1595) 的 `uci add/delete mihomo adblock_source` + `uci commit` 模式，更新 Usage [build_ipk.py:1895](build_ipk.py#L1895)）：

- `get_adblock_sources` / `add_adblock_source` / `del_adblock_source`

> **add/del 本身不重启核心**，UCI 改动由前端"应用并重启核心"按钮触发的 `prepare_config` 刷进 yaml。

**无需写更新循环**：rule-provider 自带 `interval`，核心自动周期拉取。helper.sh 仅做**首次预下载兜底**：`prepare_config` 生成 yaml 前，对启用源用 curl 预下载到 `/etc/mihomo/adblock/`，复用 `update_subscription` 的"DNS 绕过 + curl 三段失败处理"。该目录不随 ipk 升级保留，需"缺失则重下"兜底。

#### 4.2.3 配置生成 prepare_config 改造（[build_ipk.py:864-1012](build_ipk.py#L864-L1012)）

1. 读 `adblock_enabled`（仿 [build_ipk.py:884](build_ipk.py#L884) 的 `uci -q get` 模式）。
2. DNS/TUN 块之后、规则注入之前，把 `emit_adblock_rule_providers_yaml` 产出的块合并进 run_config。**去重**：`ssproxy-ad-` 前缀 + grep 跳过订阅已定义同名 key，避免 YAML 重复 key 报错。注入点 [build_ipk.py:979-980](build_ipk.py#L979-L980) 之后。
3. **rules 注入顺序调整**为：bypass → **用户白名单（access rules 的 direct，必须优先于广告 REJECT 防误杀）** → **广告 RULE-SET REJECT** → 其余 access rules → 订阅规则。

```yaml

    "rule-set:ssproxy-ad-<name>": rcode://success
```

#### 4.2.5 保存→生效链路（复用现有，无需新机制）

```
用户在 adblock.js 增删源 / 切开关
   │ fs.exec('helper.sh', ['add_adblock_source', ...])  →  uci add + commit（不重启）
   ▼
```

#### 步骤 2 · helper.sh 新增 YAML 产出函数（仿 [emit_access_rules_yaml:673](build_ipk.py#L673)）

在 `emit_access_rules_yaml` 函数之后插入两个函数：

```sh
# 产出受控 rule-providers: 块（仅启用的源）
emit_adblock_rule_providers_yaml() {
	local enabled; enabled=$(uci -q get mihomo.config.adblock_enabled || echo 0)
	[ "$enabled" = "1" ] || return 0
	local has=0
	uci show mihomo 2>/dev/null | sed -n 's/^mihomo\.\(.*\)=adblock_source$/\1/p' | while read -r sid; do
		local e name url behavior fmt
		e=$(uci -q get mihomo.$sid.enabled); [ "$e" = "1" ] || continue
		name=$(uci -q get mihomo.$sid.name); url=$(uci -q get mihomo.$sid.url)
		[ -n "$name" ] && [ -n "$url" ] || continue
		behavior=$(uci -q get mihomo.$sid.behavior || echo domain)
		fmt=$(uci -q get mihomo.$sid.format || echo yaml)
		has=1
		echo "  $name:"
		echo "    type: http"
		echo "    behavior: $behavior"
		echo "    format: $fmt"
		echo "    url: $url"
		echo "    path: ./ruleset/$name.yaml"
		echo "    interval: 86400"
	done
	# 通过返回值告知调用方是否有内容（供 prepare_config 决定是否输出块头）
}

# 产出 RULE-SET,<name>,REJECT 行
emit_adblock_rules_yaml() {
	local enabled; enabled=$(uci -q get mihomo.config.adblock_enabled || echo 0)
	[ "$enabled" = "1" ] || return 0
	uci show mihomo 2>/dev/null | sed -n 's/^mihomo\.\(.*\)=adblock_source$/\1/p' | while read -r sid; do
		local e name
		e=$(uci -q get mihomo.$sid.enabled); [ "$e" = "1" ] || continue
		name=$(uci -q get mihomo.$sid.name); [ -n "$name" ] || continue
		echo "  - 'RULE-SET,$name,REJECT'"
	done
}
```

> 函数内部不输出 `rule-providers:` 块头，块头由 `prepare_config` 在确认有内容时输出，避免空块导致 YAML 解析问题。

### 4.4 关键风险与对策

| 风险 | 对策 |
|---|---|
| rule-provider 首次下载失败致核心 fatal | `prepare_config` 用 curl 预下载启用源到 `/etc/mihomo/adblock/`（复用 update_subscription 的 DNS 绕过 + curl 三段）；目录不随升级保留，"缺失则重下" |

## 五、方案 B：AdGuard Home（对比，不采用）

### 原理
AdGuard Home 是一台独立的"DNS 黑洞"服务器，独占 53 端口，把广告/追踪域名解析为空 IP，并提供 Web 管理界面、客户端统计、DoH/DoT 加密解析。

### 方案
作为独立服务部署在路由器（或旁路由），dnsmasq 让出 53 端口、或 dnsmasq 上游指向 AdGuard Home，由它统一过滤后再转发上游 DNS。

### 为何不采用
- **与现有 DNS 劫持冲突**：本项目 `enable_dns_hijack`（[build_ipk.py:117](build_ipk.py#L117)）已把 dnsmasq 改为转发给 Mihomo :1053。再引入 AdGuard Home 抢 53 端口，三者（dnsmasq / AdGuard Home / Mihomo DNS）链路纠缠，维护成本高。
- **资源开销**：二进制 10-15 MB、运行内存 30-50 MB，对软路由是不小负担，而方案 A 几乎零开销。
- **与代理分流正交**：AdGuard Home 只做 DNS 过滤，无法像 Mihomo 规则引擎那样把"广告 REJECT、其余分流代理"统一在一套规则里。
- **违背单核心设计**：项目核心设计是"Mihomo 一个核心管一切"，引入第二个常驻服务打破这一原则。

---

## 六、方案 C：dnsmasq + hosts 黑名单（低配备选，不采用）

### 原理
dnsmasq 原生支持加载 hosts 文件，把广告域名解析到 `127.0.0.1`（黑洞）。OpenWrt 的 `adblock` / `adblock-lean` 插件即此思路。

### 方案
在 `enable_dns_hijack` 钩子里扩展：定时拉取 anti-AD hosts 列表，转换后写入 dnsmasq 配置/addn-hosts，reload dnsmasq 生效。

### 为何不采用（仅留作低配备选）
- **列表过大拖累 dnsmasq**：anti-AD 有数十万条，全量灌进 dnsmasq 使解析变慢、内存上涨，而方案 A 的规则匹配在 Mihomo 核心内、不影响 dnsmasq。
- **仅 DNS 层**：无法处理 IP 直连广告，无智能分流、无白名单优先级。
- **需自写更新机制**：方案 A 的 rule-provider 自带 `interval` 自动更新，本方案需自建拉取/转换/reload cron。
- **可挂载点已存在但收益有限**：`enable_dns_hijack` 是现成钩子，但综合收益不如方案 A。

## 七、结论与选型

**选定方案 A（Mihomo rule-provider + DNS nameserver-policy 双重拦截）**，理由：

1. **架构契合**：复用 Mihomo 核心与现有"受控配置注入"管道（prepare_config），零额外进程、零额外内存。
2. **双重拦截更彻底**：DNS 层（nameserver-policy rcode://success）+ 规则层（RULE-SET REJECT），fake-ip 架构下层层兜底。
3. **前端成本低**：独立「广告过滤」菜单页直接复制 rules.js 模式，预设源 + 自定义增删。
4. **自动更新原生支持**：rule-provider 自带 interval，无需自建 cron。
5. **rpcd acl 无需改动**：文件级授权已覆盖新 helper 子命令。

B（AdGuard Home）否决：违背单核心设计、资源开销大、与 DNS 劫持冲突。C（dnsmasq hosts）否决：列表拖累、仅 DNS 层、需自写更新，仅保留为极低配设备的理论备选。

实现说明：本方案已确认选型并据此编码。代码改动集中在 `build_ipk.py` 内嵌字符串——UCI 默认配置（`root/etc/config/mihomo`）、helper.sh 后端（`emit_adblock_*` 函数、CRUD 子命令、`prepare_config` 改造）、前端 `adblock.js`、menu.d 菜单项。构建用 `python3 build_ipk.py`，部署用 `./deploy.sh`。
