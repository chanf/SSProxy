# 链式代理：产品合理性与优化建议

> 项目：luci-app-ssproxy（水杉代理）
>
> 模块：独立「链式代理」Tab（落地节点 + 数据链路）
>
> 关联文档：[`chain-proxy-product-design.md`](./chain-proxy-product-design.md)
>
> 基线版本：1.0.0-189 前后（实现已用 `dialer-proxy` 替代已废弃的 `relay`）
>
> 性质：产品/架构评审与后续优化 backlog，**非已批准的实现规格**

---

## 1. 总评

**当前产品设计整体合理，方向正确。**

把「两跳固定链路」拆成两类业务对象：

1. **落地节点**：用户自有的第二跳代理资产  
2. **数据链路**：`设备 IP/CIDR → 订阅前置节点 → 落地节点` 的绑定关系  

相比旧版自由 `relay_chain` / `relay_bind`，更贴近真实场景（机场入口 + 自建 VPS 落地），也比让用户手写 YAML 更可控。

技术选型上：

- 使用 Mihomo 官方仍支持的 **`dialer-proxy`** 表达两跳，避免 v1.19.x 已移除的 `relay` 组类型  
- 校验失败 **只跳过并写 `mihomo-chain` 日志**，不拖垮核心启动  
- 内部代理名基于 UCI section id，展示名可改、引用不炸  

**主要问题不在模型本身，而在：**

- 与透明代理 / 白名单 / DNS 的边界未产品化  
- 「配置了」与「真正生效」的状态表达不够直白  
- 部分观测与采集实现偏重，主路径可解释性仍可加强  

不建议再改回「任意 N 跳自由列表」。后续增强应优先：**边界正确、失败可解释、协议与性能按需扩展**。

## 2. 运行模型（现状）

```text
局域网设备
    │  (nft TProxy / 白名单 ACL 决定是否进入 Mihomo)
    ▼
Mihomo rules: SRC-IP-CIDR[6],<device>,ssproxy-chain-<sid>
    │
    ▼
链路代理 ssproxy-chain-<sid>
    │  dialer-proxy: <订阅前置节点>
    ▼
订阅节点（第一跳）
    │
    ▼
落地节点副本（第二跳，参数来自 landing_node）
    │
    ▼
目标站点
```

`prepare_config` 注入顺序要点：

| 阶段 | 内容 |
| --- | --- |
| rules 前部 | 内置旁路 → **数据链路 SRC-IP 规则** → UCI 访问规则 → 广告 RULE-SET |
| proxies 后段 | 全部合法 **落地 proxies** → 每条链路一份带 `dialer-proxy` 的 **链路 proxies** |

全局统一前置保存在 `mihomo.config.chain_front_node`：

- `individual`：每条链路用自己的 `subscription_node`  
- 具体节点名：所有链路的有效前置统一为该节点，**不改写**各行 UCI 原值

## 3. 现状优点（应保留）

| 维度 | 评价 |
| --- | --- |
| 概念模型 | 用户只填设备 / 前置 / 落地，不碰 Mihomo 组类型 |
| 稳定内部名 | `ssproxy-landing-<sid>` / `ssproxy-chain-<sid>` |
| 落地引用 | 链路存 section id，不存展示名 |
| 统一前置 | 覆盖生效但不覆盖各行配置，运维友好 |
| 失败策略 | 无效配置 skip + 日志，保护主服务 |
| 配置持久化 | 完整 UCI 备份到包外，卸载重装可恢复 |
| 观测闭环 | 状态条、延时测试、红绿通讯、速率/累计流量 |
| 测试覆盖 | `tests/shell/test_chain_proxy.py` 覆盖注入、统一前置、IPv6、健康、指标、前端控件等 |

对「按设备走固定两跳」这一主场景，产品边界是清晰的，**v1 模型值得保留**。

## 4. 产品层问题与建议

### 4.1 与透明代理 / 白名单的关系不透明（P0）

**问题**

链路规则只在 Mihomo **内部**生效。若 `acl_mode=whitelist` 且设备不在 `acl_ips`：

- nft 在内核态旁路，流量**不会进入** Mihomo  
- 页面上链路显示「已配置 / 已注入」，实际完全不走链路  

DNS 劫持同理：设备不在白名单时可能拿真实 DNS，与 fake-ip 分流预期不一致。

**建议**

1. 保存 / 启用数据链路时做一致性校验：白名单模式下提示「请把该设备加入受控 IP」  
2. 提供一键「加入白名单」动作  
3. 状态条增加：链路已注入，但设备当前是否会被 TProxy 接管（可复用 `emit_tproxy_rules` / ACL 判定逻辑）  
4. 文档与页面简短说明：链式代理依赖流量先进入 Mihomo  

这是最容易被误判为「链式代理坏了」的产品缺口。

### 4.2 「保存配置」与「真正生效」两步割裂（P0）

**问题**

- 落地 / 链路改完需应用并重启，才进入 `/tmp/mihomo_run.yaml`  
- `test_landing_node` 要求「已应用」，未保存未重启时测不了  
- 统一前置变更会立刻 `set_chain_front_node` + `restart`，与表单「保存再应用」路径不一致  

**建议**

1. 未注入时禁用测试按钮，并写清原因（如「请先保存并应用」）  
2. 状态条明确区分：`UCI 已配置` / `校验通过` / `运行配置已注入`  
   - 后端 `get_chain_status` 已有 `landing_*`、`link_*`、`runtime_*` 字段，UI 可更直白  
3. 统一前置变更前弹确认：「将重启 Mihomo 服务」  
4. 链路启用数与 runtime 注入数不一致时高亮告警（并引导打开链式日志）  

### 4.3 落地节点全量注入可能污染语义（P1）

**问题**

当前所有**启用且合法**的落地节点都会写入 `proxies:`，即使没有任何数据链路引用。

影响：

- 运行配置变胖  
- 外部控制台 / 其它视图若暴露全部 proxies，会出现无用落地项  
- 用户可能误以为「添加落地 = 全局生效」  

**建议（二选一或组合）**

1. **仅注入被至少一条启用链路引用的落地**（语义更干净）  
2. 保留全量注入，但在 UI 明确文案：「落地节点仅为链路第二跳资产，不会单独接管设备」  

链路代理本身仍是落地参数 + `dialer-proxy` 的副本，与是否单独注入 `ssproxy-landing-*` 可解耦。

### 4.4 设备唯一性规则前后端不一致（P1）

**问题**

- 前端：同一设备仅允许一条**启用**链路  
- 后端：`seen_devices` 按 UCI 遍历顺序 **先到先得**，重复静默 skip  

缺口：

- 多条禁用链路可重复同一 IP；重新启用时谁生效依赖顺序，不可预期  
- 用户看不到「被跳过」的明确错误，只能翻日志  

**建议**

1. 后端强制：同一规范化 `device_ip` 仅允许一条 `enabled=1` 的 `data_link`  
2. 冲突时 `chain_log` + 接口返回明确错误，而不是静默丢弃  
3. 可选：保存时规范化单地址（补 `/32` / `/128`）再去重，避免 `1.2.3.4` 与 `1.2.3.4/32` 被视为两条  

### 4.5 协议能力边界需产品化（P2）

**现状**

首期支持 SOCKS5 / HTTP / SS / Trojan / VMess / VLESS 的常见 **TCP** 字段；不做 WS / gRPC / Reality 等自由 YAML 绕过。

**问题**

真实落地（尤其 VLESS / Trojan）常需要 WS + TLS + Reality。字段不足时用户仍可能「保存成功」，运行时才发现不可用。

**建议**

1. UI 标明「当前传输：TCP」  
2. 协议字段不完整时，后端 `landing_node_valid` 给出可读原因  
3. 按需规划扩展：`network`、`ws-opts`、`reality-opts` 等结构化字段，**不**引入自由文本 YAML  

### 4.6 观测偏重、主路径可解释性偏轻（P1–P2）

**现状优点**

红绿通讯状态、5s 速率、累计流量、链式日志弹窗。

**缺口**

- 失败原因默认藏在 `get_chain_log`，主列表不展示  
- 统一前置节点失效时，下拉有「已失效」，但有效链路数下降不够醒目  
- 缺少从 LAN 主机列表「选设备」的入口  
- 访问日志 / 仪表盘未突出 `ssproxy-chain-*` 出站，排障链路长  

**建议**

1. 每条链路旁展示最近一次 skip / test 失败摘要（可缓存于 `/tmp`）  
2. 统一前置失效时在状态条红色提示，并列出受影响链路数  
3. 访问日志对 `ssproxy-chain-*` 策略加徽标或筛选  
4. 流量面板可后置或降采样；优先保证「配得对、看得见为什么不对」  

### 4.7 文档与验收标准过时（P3）

[`chain-proxy-product-design.md`](./chain-proxy-product-design.md) 验收第 4 条仍写「两跳 **relay** 组」。

实现与 CHANGELOG 已明确为 `dialer-proxy`。应同步改为：

> `/tmp/mihomo_run.yaml` 正确包含落地 `proxies`、带 `dialer-proxy` 的链路 `proxies`、以及源 IP 规则。

## 5. 技术 / 实现层优化

### 5.1 YAML 转义（P2）

`yaml_quote` 目前主要处理换行与双引号，对反斜杠等不完整。

密码、UUID、节点名中的特殊字符可能导致非法 YAML 或注入风险。应统一为可靠转义（或「先 JSON 再生成 YAML 片段」的策略）。

### 5.2 链路校验所依据的配置文件（P1）

规则注入较早，链路 proxies 在 mixed overlay 之后写入。  
`data_link_valid` / `effective_data_link_node` 校验前置节点时依赖的 config，与最终 `run.yaml` 在 custom / mixed 模式下可能不完全一致。

**建议**：链路校验统一以「即将启动的最终 run 配置」或「与 `collect_proxy_names` 同源的最终节点集合」为准。

### 5.3 统一前置变更的重启成本（P2）

`dialer-proxy` 写在静态 proxy 对象上，改前置通常必须重写配置。

可评估：

1. Mihomo 是否支持局部热更新（能力边界需实测）  
2. 至少缩短路径为 `prepare_config` + 更轻量的 reload，而不是完整 init 重跑所有旁路逻辑  

在能力不允许前，产品层用确认弹窗管理预期即可。

### 5.4 健康判定精度（P1）

`update_data_link_health_from_connections` 对连接 JSON 使用 `grep -qF "$group"`，存在子串误匹配风险。

**建议**：用 `jsonfilter` 精确比对 `chains` / `policy` 字段（与 `resolve_data_link_sid` 对齐）。

### 5.5 流量采集性能（P2）

`collect_data_link_traffic` 对每个连接多路 `sed` / `awk` / `jsonfilter`，连接多时对低端 x86（如 J4125）不友好。

**建议**：

1. 与 `collect_connections` 共用一次解析结果  
2. 一次导出结构化行，再单次 awk 聚合  
3. 评估轮询间隔是否必须与全局流量 5s 对齐  

### 5.6 累计流量策略（P3）

累计值每 60s 写入 `/etc/mihomo/.data_link_traffic`，重置靠手动。

可选：按日清零、或与全局流量页策略对齐，并在 UI 标明统计口径。

### 5.7 工程结构（P3）

链式逻辑已在 `helper.sh` 中占较大篇幅。长期可拆为 `chain.sh` 并由 helper source，降低回归与冲突成本（用户无感知）。

## 6. 不建议做的方向

| 方向 | 原因 |
| --- | --- |
| 恢复任意 N 跳 UI | 回到旧 relay 复杂度，验收与排障成本高 |
| 用自由文本 YAML 绕过协议表单 | 破坏校验、备份与安全边界 |
| 把链式代理做成全局默认出站 | 与「按设备绑定」产品目标冲突；应用 ACL/规则即可 |
| 静默把设备自动加入白名单且不提示 | 改变用户 ACL 意图，需显式确认 |

## 7. 建议优先级汇总

| 优先级 | 项 | 类型 | 主要改动面 |
| --- | --- | --- | --- |
| **P0** | 白名单 / TProxy 与链路设备一致性校验与提示 | 产品正确性 | helper + chain.js + 可选 UCI |
| **P0** | 未注入 / 注入失败原因直接展示 | 体验 | chain.js + 可选 get_chain_status 扩展 |
| **P1** | 设备 IP 唯一性后端强制，冲突明确报错 | 可靠性 | helper + 前端校验对齐 |
| **P1** | 仅注入被引用落地，或文案澄清「落地 ≠ 全局代理」 | 语义 | prepare_config / chain.js |
| **P1** | 健康判定改为精确 JSON 匹配 | 正确性 | helper |
| **P1** | 链路校验节点集合与最终 run 配置对齐 | 正确性 | prepare_config / valid 路径 |
| **P2** | 协议扩展（WS / Reality 等）按需 | 能力 | UCI + emit + 表单 |
| **P2** | 流量采集减负 / 与全局采集合并 | 性能 | helper collect_* |
| **P2** | `yaml_quote` 加固 | 安全 | helper |
| **P2** | 统一前置变更确认与更轻量 reload | 体验/稳定性 | chain.js + init/helper |
| **P3** | 累计流量策略与文档验收条（去掉 relay 表述） | 文档/策略 | Docs + 可选 helper |
| **P3** | helper 链式逻辑拆分 | 工程 | 构建打包路径 |

## 8. 建议的验收补充（相对产品设计文档）

在现有验收标准上建议追加：

1. **ACL 联动**：白名单模式下，未纳入受控 IP 的设备配置链路时，页面有明确警告；纳入后流量命中 `ssproxy-chain-*`。  
2. **状态可区分**：仅 UCI 存在、校验失败、已注入运行配置三种状态可在 UI 区分。  
3. **冲突可解释**：同一设备两条启用链路时，保存或 prepare 给出明确错误/日志，而不是静默一条生效。  
4. **前置失效**：统一前置节点从订阅中消失后，受影响链路被 skip，状态条或列表可见，且不静默改用其它节点。  
5. **实现表述**：验收以 `dialer-proxy` 链路代理为准，不再要求 `relay` 组。  
6. **回归**：`tests/shell/test_chain_proxy.py` 全绿；关键路径在软路由上 `prepare_config` 后核心可启动。

## 9. 结论

| 问题 | 结论 |
| --- | --- |
| 产品设计是否合理？ | **合理**。固定两跳 + 落地资产 + 设备绑定，覆盖主场景且可演进。 |
| 是否还有优化空间？ | **有**，且优先集中在边界正确性与可解释性，而非再堆功能。 |
| 最大优化方向 | 把透明代理 / 白名单边界、配置生效状态、失败原因做成用户一眼能懂的产品行为。 |
| 模型是否推倒重来？ | **否**。保留现模型，按 P0→P3 迭代即可。 |
